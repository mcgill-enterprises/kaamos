# Node.js Monorepo Dependency Troubleshooting

## Diagnosis first ‚Äî always run these before touching anything

```bash
# Find all installed copies of a conflicting package and what requires them
npm why <package-name>

# Check what version is actually on disk (not what package.json says)
node -e "console.log(require('./node_modules/<pkg>/package.json').version)"

# Check if a package has a types field and exports map
node -e "
const p = require('./node_modules/<pkg>/package.json');
console.log('types:', p.types);
console.log('exports:', JSON.stringify(p.exports, null, 2));
"

# Find all nested copies of a package across workspaces
find . -path "*/node_modules/<pkg>/package.json" -not -path "*/node_modules/*/node_modules/*"
```

## Pattern: code generator / runtime library version mismatch

Tools like `buf`, `protoc-gen-es`, `openapi-generator` generate code that
imports from a specific version of a runtime library. If the installed version
doesn't match, imports will fail at compile time or runtime.

**Diagnosis:**
```bash
# Check what version the generator targets ‚Äî often in the generated file header
head -3 path/to/generated/file.ts
# e.g. "// @generated by protoc-gen-es v2.11.0"

# Check what's installed
node -e "console.log(require('./node_modules/@bufbuild/protobuf/package.json').version)"
```

**Fix:** Pin the runtime library to the version the generator targets.
For buf/protobuf-es: the generator version == the required runtime version.
```bash
npm install @bufbuild/protobuf@<generator-version>
```

**Prevent:** After any `buf generate` or similar, check the generated file
header and verify the runtime version matches. Add it to `package.json` pinned,
not with `^`.

## Pattern: package published without `types` in exports map

Some packages (notably `@bufbuild/protobuf` v2.x) ship `.d.ts` files but omit
the `types` condition from their `exports` map. TypeScript with
`moduleResolution: Node16/NodeNext/Bundler` cannot find the declarations even
though the files exist.

**Diagnosis:**
```bash
# Check if types condition is missing
node -e "
const p = require('./node_modules/<pkg>/package.json');
Object.entries(p.exports ?? {}).forEach(([k, v]) => {
  if (v && !v.types) console.log('MISSING types:', k, JSON.stringify(v));
});
"

# Verify the .d.ts files actually exist
ls node_modules/<pkg>/dist/esm/index.d.ts
```

**Fix:** Patch the package's `package.json` in a `postinstall` script.
```js
// scripts/patch-<pkg>.mjs
import { readFileSync, writeFileSync } from 'fs';
const path = 'node_modules/<pkg>/package.json';
const pkg = JSON.parse(readFileSync(path));
if (pkg.types) process.exit(0); // already patched
pkg.types = './dist/esm/index.d.ts';
for (const [key, val] of Object.entries(pkg.exports ?? {})) {
  if (val?.import && !val.types)
    pkg.exports[key] = { types: val.import.replace('.js', '.d.ts'), ...val };
}
writeFileSync(path, JSON.stringify(pkg, null, 2));
```

Add to root `package.json`:
```json
"scripts": {
  "postinstall": "node scripts/patch-<pkg>.mjs"
}
```

**Prevent:** Check a new package's `exports` map before adopting it. If it
lacks `types` conditions, budget time for the patch. File an issue upstream.

## Pattern: nested node_modules shadowing root install

In npm workspaces, if a workspace package declares a dependency that another
package in the tree also requires at a different version, npm installs a nested
copy. The nested copy shadows the root install for that workspace at runtime.

**Diagnosis:**
```bash
npm why <package>
# Look for multiple entries ‚Äî each is a separate installed copy.
# The path tells you where: packages/shared/node_modules/<pkg> = nested copy.
```

**Fix options (in order of preference):**

1. **`overrides`** ‚Äî forces the entire tree to one version:
   ```json
   // root package.json
   "overrides": { "<pkg>": "2.11.0" }
   ```
   Then delete all node_modules and lockfile and reinstall:
   ```bash
   rm -rf node_modules packages/*/node_modules package-lock.json
   npm install
   ```
   Note: overrides don't always deduplicate existing lockfile entries ‚Äî the
   full wipe is required.

2. **Remove the dependency from workspace packages** ‚Äî if a package uses a
   library only transitively, remove it from that package's `dependencies`.
   Declare it only at the root. Use `peerDependencies` in shared packages to
   document the requirement without causing npm to install a copy.

3. **Eliminate the conflicting transitive dependency** ‚Äî if package A requires
   old-version of X and you need new-version of X, consider replacing package A
   if it's not doing much. (In this codebase: `cloudevents` required
   `@bufbuild/protobuf@^1.x`; we inlined the 10 lines we needed from it.)

**Prevent:** Before adding a new dependency, check if it pulls in conflicting
versions of packages already in your tree:
```bash
npm install <new-pkg> --dry-run
npm why <known-conflict-pkg>
```

## Pattern: ts-node broken on Node 20+

`ts-node/esm` via `--loader` is deprecated and broken on Node 22+. The
`--loader` flag itself is being removed.

**Fix:** Replace `ts-node` with `tsx` in all workspace `package.json` files:
```json
"devDependencies": { "tsx": "^4.19.2" }
"scripts": { "dev": "node --watch --import tsx/esm src/index.ts" }
```

`--import tsx/esm` is the modern replacement for `--loader ts-node/esm`.

## Pattern: Kafka KRaft CLUSTER_ID must be a base64 UUID

KRaft mode rejects human-readable cluster IDs. Generate a valid one:
```bash
python3 -c "
import uuid, base64
u = uuid.uuid4()
print(base64.urlsafe_b64encode(u.bytes).rstrip(b'=').decode())
"
```

Use the output as `CLUSTER_ID` in docker-compose.yml. If you change it after
first run, do `docker compose down -v` to wipe the volume ‚Äî KRaft stores the
ID in the data directory and will reject a mismatch.

## Pattern: Kafka init container can't reach broker

If the init container uses `--bootstrap-server kafka:9092` but the broker's
`KAFKA_ADVERTISED_LISTENERS` is `localhost:9092`, the broker will advertise
itself as `localhost` back to the client ‚Äî which resolves to the client
container itself, not the broker.

**Fix:** Two listeners ‚Äî one for host access, one for inter-container:
```yaml
KAFKA_LISTENERS: PLAINTEXT_HOST://0.0.0.0:9092,PLAINTEXT_INTERNAL://0.0.0.0:29092,CONTROLLER://0.0.0.0:9093
KAFKA_ADVERTISED_LISTENERS: PLAINTEXT_HOST://localhost:9092,PLAINTEXT_INTERNAL://kafka:29092
KAFKA_LISTENER_SECURITY_PROTOCOL_MAP: PLAINTEXT_HOST:PLAINTEXT,PLAINTEXT_INTERNAL:PLAINTEXT,CONTROLLER:PLAINTEXT
KAFKA_INTER_BROKER_LISTENER_NAME: PLAINTEXT_INTERNAL
```

Init container and other Docker services use `kafka:29092`.
Host applications use `localhost:9092`.

## @bufbuild/protobuf v1 ‚Üí v2 API migration

| v1 | v2 |
|----|----|
| `import { ShipDeparted } from './navigation_pb'` | `import { ShipDepartedSchema } from './navigation_pb'` |
| `new ShipDeparted({ shipId: '...' })` | `create(ShipDepartedSchema, { shipId: '...' })` |
| `T extends Message` generic constraint | `Desc extends DescMessage` with `MessageShape<Desc>` |
| `DescMessage` from `@bufbuild/protobuf` | `DescMessage` from `@bufbuild/protobuf` (same, but not in codegenv2) |
| `toBinary(schema, msg)` | `toBinary(schema, msg)` (same) |
| `fromBinary(schema, bytes)` returns `T` | `fromBinary(schema, bytes)` returns `MessageShape<Desc>` |

Function signatures in shared helpers must change from:
```ts
function foo<T extends Message>(schema: DescMessage<T>, payload: T)
```
to:
```ts
function foo<Desc extends DescMessage>(schema: Desc, payload: MessageShape<Desc>)
```
